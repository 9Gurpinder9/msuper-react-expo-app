## 3) Roles of Main Directories

- Root
  - `backend`: Node/Express API (TypeScript). Contains `src` (authoritative TS), `dist` (compiled JS), `logs`, env example, tsconfig, package.json.
  - `frontend`: Expo app (React Native) with expo-router pages in `app`, shared code in `src`, app config files, ESLint/TS config, CI.
  - `md`: Design/architecture documentation.

- Frontend
  - `app`: expo-router routes. `_layout.tsx` defines layout/wrappers, nested folders (e.g., `super-admin`) act as route groups with their own `_layout`.
  - `src/super-admin`: feature code for Super Admin domain.
    - `components`: shared UI for the feature (e.g., `TopAppBar.tsx`).
  - `src/theme`: design system tokens and theme generator (`colors.ts`, `spacing.ts`, `typography.ts`, `index.ts` with `getTheme`).
  - `src/utils`: cross-cutting utilities like `logger`, `ToastProvider`, `network`, and `toast`.
  - `src/services`: API clients (e.g., auth service).
  - `src/hooks`: React Query-powered hooks for data flows.
  - `config`: runtime/app configuration helpers (imported via path alias `@config`).
  - `assets`: images/icons/splash assets for Expo.
  - Top-level config: `App.tsx` (router entry), `app.config.js`, `app.json`, `babel.config.js`, `metro.config.js`.
  - `.github`: workflows and PR templates; `.husky`: git hooks; `.eslintrc.js`: ESLint setup; `tsconfig.json`: TS paths/aliases.

- Backend
  - `src/app.ts`: Express app setup, core middleware (helmet, cors, json), route mounting (`/super-admin`), error handler.
  - `src/server.ts`: Entrypoint; loads env, connects Redis, starts server.
  - `src/routes`: top-level or aggregator routes (`routes.ts`).
  - `src/super-admin`:
    - `controllers`: route handlers (e.g., `superAdmin.controller.ts`).
    - `routes.ts`: feature-specific routing.
  - `src/middleware`: `authenticate.ts` (JWT guard), `errorHandler.ts` (centralized error responses + logging), `requestId.ts`, `rateLimit.ts`, `validate.ts`.
  - `src/utils`: `logger` (winston), `generateJwt`, `emailSender`, `otpGenerator`, `telegramSender` (supporting services).
  - `src/config`: validated environment configuration via Joi.
  - `src/database`: `redisClient` (connection/bootstrap), `supabaseClient` (Supabase SDK init).
  - `dist`: compiled JS mirror of `src`.
  - `logs`: persisted log files via winston.
  - `.env.example`: environment template (PORT, SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY).

## 4) Coding Conventions (Inferred)

- General
  - TypeScript-first across both frontend and backend.
  - Single quotes and semicolons standard in checked files.
  - Feature-first structure for вЂњSuper AdminвЂќ across FE/BE.

- Frontend (Expo/React)
  - Routing via expo-router; `app` contains route files, `_layout.tsx` wrappers, and nested route groups (`app/super-admin`).
  - Component naming: React components in PascalCase (`TopAppBar.tsx`), route files tend to be kebab-case (e.g., `otp-verify.tsx`).
  - Theming with `react-native-paper` + custom tokens in `src/theme`; `getTheme(scheme)` provided and consumed in root layout.
  - Utilities are colocated in `src/utils` and exposed via providers/hooks (`ToastProvider`, `logger`).
  - Path aliases in `tsconfig.json`: `@theme`, `@config`, `@super-admin/*`, `@utils/*`.
  - ESLint: `eslint:recommended`, `plugin:react/recommended`, and `plugin:react-hooks/recommended`. No Prettier config committed.
  - Husky + lint-staged enforce linting on staged TS/JS before commit.

- Backend (Node/Express)
  - Express 5 with middleware ordering: security (helmet), CORS, JSON parsing, routes, final error handler.
  - Controller/route split per feature (`super-admin/controllers` + `super-admin/routes.ts`).
  - Auth middleware (`authenticate.ts`) uses Bearer JWT (вЂњverifyJwtвЂќ from `utils`).
  - Centralized error handling (`errorHandler.ts`) logs and returns normalized JSON.
  - Logging via Winston with both console and file transports writing to `logs/app.log`.
  - `tsconfig.json` targets `es2020`, `commonjs` modules, `rootDir: src`, `resolveJsonModule: true`.
  - Environment loading via `dotenv` in `server.ts`.

## 6) Codex Session Logging (Required)

Purpose

- Persist a brief, human-readable summary of every change Codex makes on your behalf during a session.

Where to write logs

- Location: repository root under a fixed directory: `update-log/`.
- Structure: `<repo-root>/update-log/<timestamp> <relevant-name>.md` (no subfolders per feature).
- Create the `update-log` directory if it does not exist.

Filename format

- Timestamp: `dd-MM-YYYY_hh-mm-ss-AM/PM` (12-hour clock with AM/PM).
- Use dashes in time instead of colons for cross-platform file safety.
- Example: `05-11-2025_03-42-10-PM super-admin.md`.
- All log files use the `.md` extension.

What to log (keep it concise)

- A one-sentence summary of what changed.
- A compact list of operations with file paths and the type of action:
  - Added/Updated/Removed: `<path>` вЂ” short note
  - Installed/Uninstalled: `<package@version>` вЂ” command used (if any)
- Avoid long diffs or verbose explanations; this is a summary, not a changelog.

Template (use for each session)

```
# Codex Session Log вЂ” <relevant-name>
- When: <dd-MM-YYYY hh:mm:ss AM/PM>
- Summary: <1вЂ“2 lines>
- Operations:
  - Added: <path> вЂ” <short note>
  - Updated: <path> вЂ” <short note>
  - Removed: <path> вЂ” <short note>
  - Installed: <pkg@ver> вЂ” <command>
  - Uninstalled: <pkg@ver> вЂ” <command>
- Notes: <optional, keep brief>
```

Operational rules

- One log file per Codex session acting on instructions.
- Ask for the "relevant name"; if not provided, default to `general`.
- Always include paths relative to repo root (no absolute paths).
- Write logs to `update-log/` only; do not create feature-named directories.
- If no changes were made, no log is created.

Scope

- Applies to code/asset edits, file adds/removals, configuration changes, and dependency installs/uninstalls performed by Codex.

Non-goals

- Do not include full diffs or detailed implementation notes; keep it high-level and scannable.
